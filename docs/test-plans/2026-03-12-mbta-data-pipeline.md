# MBTA Data Pipeline — Human Test Plan

## Prerequisites

- Local dev server running: `python -m http.server 8000` from project root
- Valid `config.js` with real MBTA API key
- `data/mbta-static.json` exists (generated by `node scripts/fetch-mbta-data.mjs`)
- All automated tests passing:
  ```
  node tests/polyline-merge.test.js
  node tests/static-data.test.js
  node tests/fetch-mbta-data.test.js
  ```

## Phase 1: Static Data File Integrity

| Step | Action | Expected |
|------|--------|----------|
| 1.1 | Run `node -e "const d=require('./data/mbta-static.json'); console.log('routes:', d.routes.length, 'stops:', Object.keys(d.stops).length)"` | ~177 routes, ~7600 stops |
| 1.2 | Inspect `routes` array for Red Line (type 1), Green-B (type 0), CR-Fitchburg (type 2), Route 1 bus (type 3), Hingham Ferry (type 4) | Each route has `id`, `type`, `color`, `polyline`, `directionNames`, `directionDestinations` |
| 1.3 | Spot-check a subway route polyline (e.g., Red Line) | Single merged polyline array (not separate inbound/outbound) |

## Phase 2: Browser Hydration from Static Data (AC2.1, AC2.2, AC2.3)

| Step | Action | Expected |
|------|--------|----------|
| 2.1 | Open DevTools > Application > Local Storage. Clear `ttracker-static-data` key. Clear cache. | localStorage empty for this key |
| 2.2 | Navigate to `http://localhost:8000`. Open Network tab (filter Fetch/XHR). | Page loads with map visible |
| 2.3 | Confirm request to `data/mbta-static.json` (fresh visit, AC2.1) | Request present, status 200 |
| 2.4 | Check Application > Local Storage for `ttracker-static-data` key | Key present with `version: 1` and `data` object |
| 2.5 | Reload. Check Network tab. | No request to `data/mbta-static.json` (AC2.2). One `/routes?fields[route]=id` staleness check. |
| 2.6 | Verify subway polylines: Red, Orange, Blue, Green lines visible | Single merged lines (not doubled) |
| 2.7 | Verify stop markers appear along subway routes | Circular markers at station locations |
| 2.8 | Verify route panel lists all types: Subway, Bus, Commuter Rail, Ferry | All four categories present |
| 2.9 | Confirm no startup requests to `api-v3.mbta.com` for routes/stops | Only SSE + one `/routes?fields[route]=id` |

## Phase 3: Staleness Check (AC3.1 - AC3.4)

| Step | Action | Expected |
|------|--------|----------|
| 3.1 | Check Network tab for `/routes?fields[route]=id` | Exactly one request shortly after load |
| 3.2 | If IDs unchanged, confirm no subsequent `data/mbta-static.json` fetch | No re-fetch (AC3.2) |
| 3.3 | Edit localStorage to remove a route from cached bundle, reload | Staleness check detects mismatch, re-fetches file, updates localStorage (AC3.3) |
| 3.4 | Disconnect network after load, reload | Loads from cache. Staleness check fails silently (AC3.4) |

## Phase 4: Fallback Behavior (AC2.4)

| Step | Action | Expected |
|------|--------|----------|
| 4.1 | Clear localStorage. Rename `data/mbta-static.json` to `.bak`. Reload. | Console warning about failed static data load |
| 4.2 | Verify fallback to live API | Network shows MBTA API requests for routes/stops. Map renders correctly. |
| 4.3 | Restore file (rename `.bak` back) | File restored |

## Phase 5: Build Pipeline (AC5.1, AC5.2)

| Step | Action | Expected |
